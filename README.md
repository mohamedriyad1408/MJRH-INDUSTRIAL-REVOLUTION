# MJRH — نظام إدارة وتشغيل المغاسل

آخر تحديث توثيقي: 2026-06-26

هذا الملف هو مرجع المشروع حتى لا يبدأ أي شات جديد من الصفر. اقرأه أولًا قبل أي تعديل جديد.

---

## 1) هدف المشروع

بناء نظام إدارة مغاسل كامل وسهل لصاحب العمل غير التقني، بحيث يدخل أقل بيانات ممكنة والنظام يقوم بالباقي:

- تشغيل الطلبات من الاستقبال حتى التسليم.
- إدارة القطع والتكويد والمحطات.
- إدارة المندوبين والخريطة والاستلام والتوصيل.
- إدارة العملاء والبوابة العامة وتتبع الطلب.
- إدارة الخزن والتحصيلات والمصروفات والرواتب.
- إنشاء القيود المحاسبية تلقائيًا من العمليات اليومية.
- إقفال الخزن اليومية بسهولة.
- تقارير تشغيل ومالية وتنبيهات ومركز متابعة يومي.
- دعم أكثر من فرع داخل نفس المغسلة مع فصل بيانات الفروع وتجميعها للمالك/المدير فقط.

المبدأ الأساسي:

> صاحب المغسلة لا يحتاج يفهم محاسبة. يكتب عملية بسيطة مثل “اشتريت مسحوق” أو “حصّلت طلب”، والنظام يحولها لحركة خزنة وقيد محاسبي وتقرير تلقائيًا.

## 2) القاعدة المنطقية للفروع والمغاسل

### المغسلة / Tenant
تمثل شركة أو براند أو مالك مستقل.

مثال:

```txt
Dry Tech
```

### الفرع / Branch
يمثل نقطة تشغيل داخل نفس المغسلة.

مثال:

```txt
Dry Tech
- الفرع الرئيسي
- The Fount Mall
- Main
- فرع البنفسج
```

### متى ننشئ مغسلة جديدة؟
فقط إذا كان هناك:

- مالك مختلف.
- شركة أو براند مستقل.
- اشتراك منفصل.
- حسابات وخزن وتقارير منفصلة بالكامل.
- لا يجب أن يرى مالك الأولى بيانات الثانية.

### متى ننشئ فرعًا؟
إذا كان نفس المالك أو نفس البراند ويحتاج فصل تشغيل كل نقطة مع تجميع عام للمالك.

## 3) قاعدة الصلاحيات المطلوبة للفروع

- المالك Owner:
  - يرى كل الفروع.
  - يرى التجميع العام.
  - يدخل شاشة أي فرع.
  - يرى الحسابات والقيود والخزن والتقارير العامة.

- مدير التشغيل / مدير خدمة العملاء:
  - يرى كل الفروع حسب صلاحيات الإدارة.
  - يراجع التشغيل والتقارير والتنبيهات.

- الموظف / المندوب:
  - يرى بيانات فرعه فقط في الجداول المرتبطة بالفرع.
  - لا يرى بيانات الفروع الأخرى.

تم إضافة دوال وصلاحيات مبدئية في قاعدة البيانات لتحقيق هذا الفصل:

- `current_employee_branch_id()`
- `can_access_branch(tenant_id, branch_id)`
- سياسات RLS على الجداول الأساسية المرتبطة بالفرع.

## 10) خطة مراجعة التشغيل من التشغيل العادي إلى تشغيل الفروع

### المرحلة A — تشغيل عادي على فرع واحد
راجع يوميًا:

1. إنشاء طلب جديد.
2. هل الطلب أخذ الفرع الصحيح؟
3. هل القطع اتولدت بعدد صحيح؟
4. هل محطة الاستقبال ترى الطلب؟
5. هل الغسيل/الكي/الجودة تعمل بترتيب منطقي؟
6. هل الدفع أثر على الخزنة؟
7. هل القيد المحاسبي اتولد؟
8. هل إقفال الخزن منطقي؟
9. هل مركز اليوم يعرض المشاكل؟
10. هل التنبيهات مفيدة وليست مزعجة؟

### المرحلة B — تشغيل فرعين
راجع:

1. موظف الفرع A لا يرى بيانات الفرع B.
2. طلب الفرع A يظهر في شاشة الفرع A فقط.
3. خزنة الفرع A منفصلة عن خزنة الفرع B.
4. مصروف الفرع A لا يظهر ضمن فرع B.
5. المالك يرى كل الفروع.
6. المدير يرى التجميع.
7. التحويل بين الخزن لا يظهر كدخل أو مصروف.
8. إقفال كل الخزن يوضح التحويلات الداخلية منفصلة.

### المرحلة C — تشغيل كل الفروع
قبل فتح كل الفروع، تأكد من:

1. كل الموظفين لهم فرع.
2. كل الخزن لها فرع.
3. كل الطلبات الجديدة لها فرع.
4. كل المصروفات الجديدة لها فرع.
5. شاشة كل فرع تعرض أرقامًا صحيحة.
6. تقرير مقارنة الفروع موجود أو على الأقل شاشة الفرع كافية مؤقتًا.

## 14) مبدأ نهائي

كل عملية في النظام يجب أن تجيب على الأسئلة التالية:

- تمت في أي مغسلة؟
- تمت في أي فرع؟
- تخص أي عميل؟
- تخص أي طلب أو قطعة؟
- أثرت على أي خزنة؟
- أنشأت أي قيد محاسبي؟
- ظهرت في أي تقرير؟
- هل تحتاج إشعارًا؟

إذا لم تجب العملية على هذه الأسئلة، فهي غير مكتملة منطقيًا.
